# 香港交通预测与港铁延误预警系统 - 综合展示指南

## 1. 总体概览

### 1.1 项目是干什么的？
本项目是一个基于大数据处理与深度学习技术的**香港全城交通智能监控与预警系统**。它如同一个“城市交通大脑”，不仅能够监控地面的路况，还能监控地下的轨道交通网络，为城市管理者和普通市民提供未来的交通延误预判。

### 1.2 目前实现了哪些核心功能？
系统目前由两个无缝集成的子模块构成，并统一在一个支持动态交互的 Web Dashboard 中展示：
1. **道路交通拥堵预测 (Road Traffic System)**：
   - 实时监控全港数千个主要干道路段。
   - 预测未来 30 分钟内各路段发生拥堵（车速 < 30km/h）的持续时间。
   - 在地图上通过绿、黄、橙、红四种颜色直观渲染路况。
2. **港铁延误风险与影响预测 (MTR Delay System)**：
   - 实时监控港铁 10 条主干线路、100 多个车站的下一班车等候时间（UP/DOWN）。
   - **初级预警**：预测每个车站未来 10 分钟内发生延误事件的概率。
   - **高级预警**：一旦系统判定延误发生，立即预测该延误的**持续时间**以及**受波及的列车班次数**，并在监控大屏上触发红色闪烁警报。

## 2. 数据来源与处理流水线

本项目的血液全部来自于真实世界的政务公开数据。我们构建了一套全自动的抓取、清洗与聚合流水线。

### 2.1 数据来源
所有原始数据均源自香港政府开放数据平台（[DATA.GOV.HK](https://data.gov.hk/sc/)）。

- **地铁数据 (MTR)**
  - **来源名称**：港铁实时列车服务资讯
  - **来源链接**：[https://data.gov.hk/sc-data/dataset/mtr-data2-nexttrain-data/resource/47f9deb9-4fee-4b67-b07d-e0bdb146ffd1](https://data.gov.hk/sc-data/dataset/mtr-data2-nexttrain-data/resource/47f9deb9-4fee-4b67-b07d-e0bdb146ffd1)
  - **数据特征**：提供各线路最多四班即将到站列车的抵达时间（ttnt）、当前时间戳以及官方的延误标记（`isdelay`）。数据频率最高为每 10 秒更新一次。
  
- **公路数据 (Road)**
  - **来源名称**：主要干道及道路交通数据
  - **来源链接**：[https://data.gov.hk/tc-data/dataset/hk-td-sm_4-traffic-data-strategic-major-roads](https://data.gov.hk/tc-data/dataset/hk-td-sm_4-traffic-data-strategic-major-roads)
  - **数据特征**：以 XML 格式提供各个 `segment_id` 的实时平均车速，每 2-5 分钟更新一次。

### 2.2 数据采集与预处理
- **采集端**：我们编写了基于 `aiohttp` 的异步网络爬虫（Data Logger），严格控制请求频率以防止被反爬虫机制封禁，24小时不间断地将快照数据（JSON/XML）落盘。
- **预处理 (特征工程)**：
  - **时空对齐**：将离散的时间戳提取出“小时”、“星期几”、“是否周末”、“是否早晚高峰”等周期性特征。
  - **标签提取**：对于公路，通过聚合连续的车速计算拥堵持续分钟数；对于地铁，使用“反向滑动窗口（Rolling Window）”技术，探测未来 10 步内是否出现 `isdelay="Y"`，从而生成二分类的风险标签。
  - **数据增强 (Data Augmentation)**：由于地铁的严重延误属于“稀有事件”，我们采用添加高斯噪声 (Gaussian Noise) 的方式对少数样本进行扩充，解决了样本不平衡问题。

## 3. 模型训练与深度学习原理

本项目在模型层抛弃了传统的统计学方法，全面拥抱深度学习（PyTorch），并利用 GPU (如 RTX 5060 Ti) 实现了高效训练。

### 3.1 核心算法：GRU (Gated Recurrent Unit)
无论是公路还是地铁，交通数据本质上都是**时间序列数据**。
- 我们采用了 GRU 门控循环单元，它是 LSTM 的改良版。GRU 能够通过“更新门”和“重置门”有效地记住长期的历史交通模式（例如过去一小时的车速衰减趋势），同时过滤掉短暂的噪声。
- **序列化输入**：模型并不只看“当前这一分钟”的数据，而是接收一个长度为 10~12 的连续时间步矩阵作为输入，这让模型具备了时间维度的系统级视野。

### 3.2 多任务学习设计
在 MTR 模块中，我们将预测拆分为两个独立的神经网络：
- **初级网络 (分类)**：单向 GRU + Sigmoid 激活函数，使用二元交叉熵损失 (`BCELoss`)，负责敏锐地嗅探延误**是否**会发生（输出 0~1 的概率）。
- **高级网络 (回归)**：多输出 GRU + ReLU 激活函数，使用均方误差 (`MSELoss`)。仅在延误发生时激活，预测延误**持续多久**以及**影响多大**，保证了物理意义上的非负性。

## 4. 前后端技术栈与实时交互机制

本项目采用了前后端分离的现代化 Web 架构。

### 4.1 后端技术：FastAPI + APScheduler
- **FastAPI**：基于 ASGI 的高性能 Python Web 框架，提供了极低的延迟和自带文档的 RESTful API 接口（如 `/predictions` 和 `/mtr/delay-prediction`）。
- **后台定时任务 (APScheduler)**：
  - 后端并非在用户请求时才去计算，而是通过一个**异步调度器**，每隔 15 秒（地铁）和 5 分钟（公路）主动向香港政府接口发起请求，拉取实时数据。
  - 数据拉取后，立即在内存中进行 Spark ETL 清洗和 PyTorch 推理，然后更新到全局缓存字典中。
  - **思想优势**：这是一种典型的“读写分离”和“缓存穿透保护”思想，无论前端有多少个用户同时访问，后端的计算压力恒定不变。

### 4.2 前端技术：Leaflet.js + 动态轮询
- **Leaflet 地图引擎**：轻量级且强大的开源 JavaScript 地图库。
- **轮询拉取 (Polling)**：前端 `map.js` 使用 `setInterval`，定期使用 `fetch()` 抓取后端的 JSON 接口，用最新的预测数据重新给地图上的矢量图形着色，实现了页面的**无刷新动态监控**。

## 5. 地铁线网的可视化重构

在开发过程中，如何将冷冰冰的 MTR API 文本数据映射到地图上是一个巨大的挑战。

- **问题**：香港政府并未直接提供轻量级的前端地铁线网 GeoJSON 文件。如果仅凭逻辑生成坐标，会导致所有地铁站“一字排开”，毫无地理参考价值。
- **解决路径**：
  1. 我们从开源社区（`mtr-map-master`）获取了一份基于 SVG 像素坐标系的港铁数据集。
  2. 我们编写了 Python 脚本 (`process_mtr_csv.py`)，运用了**仿射变换 (Affine Transformation)**，将无意义的 SVG 像素点映射、缩放、平移到了香港真实的经纬度边界 (EPSG:4326) 内。
  3. 我们为每条线路绑定了港铁官方的主题色（如荃湾线红、东涌线橙），并生成了包含 `LineString` 和 `Point` 的标准 `mtr_network.geojson`。
- **成果**：这使得前端可以利用 Leaflet 精准地将港铁网络悬浮在 OpenStreetMap 真实底图上方，结合后端的实时预警，做到了真正意义上的“地理空间情报 (Geospatial Intelligence)”。

## 6. 大数据思维与 Spark RDD 架构 (⭐ 重点)

本项目之所以具有强大的工业级落地能力，在于我们不仅会写 AI，更融入了**大数据处理思维**。

### 6.1 为什么引入 Spark？
在实际的城市级监控中，每秒钟产生的数据量（如全港数万个检测器的 XML 节点）是极其庞大的。传统的 Pandas 在处理单机大文件时会遇到内存瓶颈。

### 6.2 RDD (Resilient Distributed Dataset) 架构的应用
在我们的推理管道 (`spark_etl.py`) 中，我们引入了 PySpark：
- **分布式解析**：我们将实时拉取到的海量 XML/JSON 文本，转化为 Spark 的 RDD。
- **Map-Reduce 范式**：使用 `flatMap` 算子，将嵌套复杂的 JSON 树形结构“压平”，提取出我们需要的核心字段（如 `segment_id` 和 `speed`，或 `up_ttnt`）。
- **优势**：
  - 目前在本地，Spark 以 `local[*]` 模式运行，榨干了多核 CPU 的并行计算能力，使解析速度呈指数级提升。
  - **无缝云端扩展**：我们的系统是 Config-Driven（配置驱动）的。只需在配置文件中将 `master` 改为 AWS EMR 等真实集群的 IP，同一套代码就能立即调度成百上千台服务器并行处理全城数据，真正实现了**大数据思维的落地**。

### 6.3 优雅的硬件降级机制 (Graceful Degradation)
在工程实践中，我们考虑了部署环境的多样性。在代码加载 PyTorch 模型时，系统会主动试探 CUDA（GPU）是否可用。即使遇到服务器没有 GPU，或是遇到如 RTX 5000 系列驱动不兼容等问题，代码会**自动捕获异常，并无缝降级到 CPU 推理**。这保证了数据流水线的坚不可摧。

## 7. 本项目的优势与深远意义

1. **民生与经济价值**：
   - 提前 30 分钟的拥堵预判，能帮助物流车队和救护系统优化路线，降低城市碳排放。
   - MTR 延误的实时大屏，能帮助港铁调度中心快速定位事故波及范围，启动应急巴士接驳预案。
2. **极高的系统鲁棒性 (Robustness)**：
   - 具备从数据抓取、缺失值插补、稀有事件增强到 GPU 降级容错的一整套防御性编程体系。
3. **架构的先进性**：
   - 将“Spark 分布式数据处理引擎”、“PyTorch 深度学习引擎”与“FastAPI 异步并发微服务”三大前沿技术栈完美融合，是一套完全符合现代云原生标准 (Cloud-Native) 的工业级数据产品架构。
